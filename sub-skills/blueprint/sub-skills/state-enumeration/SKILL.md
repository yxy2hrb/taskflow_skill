# Sub-skill 2：状态枚举 Agent

## 定位

蓝图生成第二步。基于 Sub-skill 1 确认的 User Story，枚举任务流 happy-path 页面状态，确保主路径导航完整性和必要中间态覆盖。

## 输入

- `user_story`：Sub-skill 1 确认的四维度 User Story
- `brief`：原始任务流描述
- `page_dsl`：页面 DSL（辅助判断现有页面结构）

## 输出协议

Phase 2 拆成 `generate` 与 `confirm` 两步。

### generate：写入 `phase2_ask.json`

按任务流顺序排列的候选状态清单，每条含：

- `id`：`state_1 / state_2 / ...`（后续 Phase 3 引用）
- `label`：状态简称（6–14 字），只写状态名，不写长描述
- `description`：该状态的具体细节，必须包含「触发条件 / 展示信息 / 继承信息」三段
- `basis`：该状态设计所参考的知名产品设计思路或典型页面，用一句话简要说明

`phase2_ask.json` 示例：

```json
{
  "action": "ask",
  "phase": 2,
  "questionText": "以下是 happy-path 状态清单，请确认保留哪些、是否需要补充。",
  "multiSelect": true,
  "allowCustom": true,
  "note": "默认全部保留；取消不需要的状态；补充栏可写新状态。",
  "options": [
    {
      "id": "state_1",
      "label": "初始页面状态",
      "description": "触发条件：无，任务流起点。\n展示信息：...\n继承信息：无。",
      "basis": "参考 Apple Store 与淘宝首页的商品浏览起始结构，作为任务流入口。",
      "default": true
    }
  ]
}
```

### confirm：写入 `phase2_confirmed.json`

```json
{
  "action": "confirmed",
  "phase": 2,
  "states": [
    {
      "id": "state_1",
      "label": "初始页面状态",
      "description": "触发条件：...\n展示信息：...\n继承信息：无。"
    }
  ]
}
```

confirm 规则：

- 只保留用户勾选或编辑后的 state。
- 终端交互时只需输入保留状态的数字编号，多个编号使用逗号分隔。
- 选择后先展示完整状态序列；修改时输入自然语言意见，由模型重新生成完整状态序列。
- 用户未选择的原候选状态不会自动恢复，但用户可以通过修改意见新增、删除或调整状态顺序。
- 修改后必须按流程顺序连续编号为 `state_1...state_N`，并同步更新描述中的状态引用。
- `state_1` 不可删除，且必须保持第一项。
- `states.length >= 4`。
- confirmed 中移除 `basis`。

`basis` 规则：

- 由模型生成，一句话简要说明即可，不写复杂分析。
- 参考与当前 state 相关的知名产品、公司设计体系或典型页面，例如 Apple、淘宝、Google、
  微信、Amazon、京东、支付宝等，但不限于这些产品。
- 同时写清参考对象和可借鉴的设计思路，例如：
  `参考 Apple Store 商品详情页的信息分区与淘宝详情页的固定操作栏。`
- 参考必须贴合当前状态，避免只罗列品牌名称。
- 使用字段名 `basis`，不要输出 `rationale`。

`label` 约束：

- 必须是短状态名，建议 6–14 字
- 禁止包含 `·`、`：`、冒号后的长描述
- 示例：`初始页面状态`、`创建项目集页`、`项目添加状态`、`提交中状态`、`成功提示状态`

`description` 必须使用以下格式：

```text
触发条件：...
展示信息：...
继承信息：...
```

三段说明：

- `触发条件`：说明该 state 如何到达。`state_1` 写“无，任务流起点”。
- `展示信息`：说明该 state 屏幕上稳定可见的内容、区域、字段、按钮、提示文案。
- `继承信息`：说明视觉上继承了哪个已有 state 的哪些内容，或跳转到某个 state 后改变了什么(例如跳转回主页，并提示创建成功，则这就是跳转state1)；若没有则写“无”。

`继承信息` 参考蓝图阶段的 `last_state` 思路：选择视觉骨架最接近、改动量最小的已有 state。常见写法：

- `继承 state_1 的状态栏；主体区域切换为创建项目集全屏表单。`
- `继承 state_2 的全屏页骨架、状态栏和底部 Tab；在表单区域新增项目列表/项目选择内容。`
- `继承 state_3 的表单内容和底部确认按钮；按钮进入 loading，页面显示提交中反馈。`
- `跳转回state_1;在顶部新增“创建成功”Toast。`
- `无，任务流起点。`

注意，当提到全屏页，新页面，则在这里可以考虑只继承“状态栏”，可能无需继承导航栏等公共组件

## 核心规则

### 必须包含的状态类型（happy-path only）

1. **初始态**（state_1 固定，原始 HTML 快照）
2. **主路径交互态**：主流程中必须可见的弹窗、表单、配置页、详情页等
3. **必要进行中/过渡态**：若 brief 明确包含提交、创建、加载、等待结果，必须独立建 loading / progress / spinner 态
4. **已提交/提交完成态**：表单或创建动作完成提交后的稳定反馈态，例如 Toast、Snackbar、成功页、回到来源页并展示成功提示
5. **成功终态**：至少一个 happy-path 终态

### 基于 Acceptance Criteria Steps 枚举

必须根据 Phase 1 的 `acceptance_criteria_steps` 从开始到结束枚举任务流状态：

- `given/and` 中描述的初始页面、入口、已有上下文 → 通常归入 `state_1`
- `when` 触发一次用户动作后，如果画面发生稳定可见变化 → 应产生新 state
- `then/and` 中出现的可见页面、弹窗、表单、列表、loading、Toast → 应映射为 state
- 多个连续动作若发生在同一稳定画面内，可合并为同一 state，但 description 必须写清动作与展示信息
- 若 `acceptance_criteria_steps` 中有多轮 `When → Then → When → Then`，必须保持顺序，不能跳过中间可见状态

不主动生成失败态、异常态、取消态、重试态。即使 brief 同时描述“成功时...”和“失败时...”，也默认只选成功分支作为 happy-path，失败分支不得进入状态清单。只有当用户目标本身就是处理失败/取消结果时，才允许作为 happy-path 的一部分列出。

### 导航完整性（重点）

若 brief 中出现"进入 B 页"、"跳转到 Y 页"、"返回 N 页"等导航语句：

> **A 来源页 + B 目标页 + 中间过渡态 各自必须独立开 state**

禁止把 A → B → C 三个独立 UI 状态折叠成一步。

示例：`brief="在工作台点击管理应用进入管理应用页，再点击新增弹窗"` → 必须列出：

1. 工作台初始态
2. 管理应用页（列表）
3. 新增弹窗

**不能把第 2 步跳过。**

### 视觉中间态（happy-path 命中其一即应独立建态）

- 有遮罩层、弹窗、抽屉、面板展开
- 有 loading / progress / spinner / 禁用态按钮
- 有键盘弹出、输入焦点变化
- 有"确认后等待结果"的明显过渡画面

### 场景硬约束

**筛选/过滤场景**（brief 包含"筛选/过滤 + 确认后加载"）：
happy-path 至少需要四态：① 列表初始态 ② 筛选面板展开态 ③ 筛选中/加载态 ④ 加载完成结果态

**表单创建/提交场景**（brief 包含"填写/输入 + 提交/创建"）：

必须至少包含以下状态，且顺序不能合并或跳过：

1. `表单初始态`：进入表单页，字段为空或待选择，提交按钮可处于禁用/默认态。
2. `已填写可提交态`：名称/字段/选项/项目等必要信息已填充，提交按钮明确可点击。
3. `提交中/loading 态`：用户点击“确认/提交/创建”后，请求处理中；按钮进入 loading / disabled，或页面出现 progress / spinner / 提交中文案。
4. `已提交/提交完成态`：系统校验通过并提交完成后的稳定反馈，如跳转回来源页并展示 Toast/Snackbar、进入成功页、展示“创建成功/提交成功”提示。

硬约束：

- `提交中/loading 态` 必须独立成 state，不得只写在 `已提交/成功提示态` 的触发条件里。
- `已提交/提交完成态` 必须独立成 state，不得与 `提交中/loading 态` 合并。
- 若 brief 写的是“点击确认后顶部弹出成功提示”，也必须拆成“点击确认后的提交中态”与“创建成功提示态”两个 state。
- `已填写可提交态` 可以承接表单填写过程，但必须在 description 中明确字段已填写、选项已选择、提交按钮可点击。

**输入法场景**（brief 明确提到"软键盘/keyboard"）：
必须有独立"软键盘弹出态"，即使系统自动唤起也不得省略

## LLM 输出协议

```json
{
  "action": "ask",
  "phase": 2,
  "questionText": "...",
  "options": [
    {
      "id": "state_1",
      "label": "初始页面状态",
      "description": "触发条件：无，任务流起点。\n展示信息：我的工作台页面，包含任务入口和原始页面骨架。\n继承信息：无。",
      "basis": "参考 Apple Store 与淘宝首页的商品浏览起始结构。"
    },
    {
      "id": "state_2",
      "label": "主路径中间状态",
      "description": "触发条件：用户执行对应操作。\n展示信息：展示当前步骤要求的表单、弹窗、列表或详情内容。\n继承信息：继承 state_1 的状态栏、导航栏或底部 Tab；主体区域切换为当前步骤内容。",
      "basis": "参考淘宝购物车确认浮层的清单核对模式。"
    },
    ...
  ],
  "multiSelect": true,
  "note": "默认全部勾选即可；如果觉得某个状态没必要，取消它；如需补充其它状态请在补充栏写。"
}
```

## 约束

- `options >= 4` 条
- 按时间先后排序，`state_1` 一定是初始态
- 每个 option 必须包含 `id / label / description / basis`
- `label` 只写简称，不得包含长描述
- `description` 必须包含「触发条件：」「展示信息：」「继承信息：」三段
- 只列 happy-path 状态；不主动列失败态、异常态、取消态、返回态
- 若 brief 同时出现成功和失败结果，默认选择成功路径，禁止输出“失败状态/重试状态”
- 允许必要的动画/过渡过程描述，但不要用动效代替状态本身
- 输出严格 JSON
